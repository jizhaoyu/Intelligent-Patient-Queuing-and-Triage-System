# 患者智能排队分诊系统｜面试速记版

## 一句话介绍

这是一个面向医院门诊/急诊场景的智能排队分诊系统，我做的重点不是几个 CRUD 页面，而是把 **患者自助机取号、分诊评估、自动入队、医生叫号、屏显联动、异常治理** 串成了一条真实业务闭环。

---

## 30 秒版本

> 这是一个医院门诊/急诊场景的智能排队分诊系统。患者可以在院内自助机完成取号，系统会自动建档或识别患者、创建就诊、生成分诊结果并自动入队；医生工作台可以叫号、复呼、过号和完成接诊；屏显端会同步展示候诊状态；后台还能处理“已分诊未入队”异常。AI 已接入分诊建议，但只是辅助层，不会阻断主链路。

---

## 1 分钟主线

> 我这个项目最核心的是把多端和多状态打通。
>
> 患者端支持已有患者核验取号，也支持新患者现场建档取号。提交后，系统会先判断有没有有效排队，如果已经在排队就直接返回当前结果；如果没有，就自动创建 visit、自动到诊、生成分诊 assessment、自动入队，并返回票号、排位、预计等待时间和下一步引导。
>
> 工作台这边，医生可以对当前诊室患者执行叫下一位、复呼、过号、完成接诊，还可以设置下次优先复诊。屏显端会按“排队中”和“候诊中”分开展示。后台则能识别“已分诊未入队”的异常并补录修复。
>
> 所以这个项目的重点不是页面多，而是主链路、状态流转、多端协同和异常治理都落地了。

## 必背 5 个亮点

1. **主链路跑通**
   患者自助机取号 -> 自动建就诊 -> 分诊 -> 自动入队 -> 医生叫号。

2. **多端协同**
   患者端、工作台、屏显端、后台围绕同一套业务状态联动。

3. **双状态体系**
   `visit` 管就诊生命周期，`queue` 管排队和叫号过程。

4. **异常治理**
   已分诊未入队可以被识别出来，并由管理员一键补录修复。

5. **AI 可用但不硬依赖**
   AI 给建议和风险提示，失败时自动回退到规则结果。

---

## 必背 3 条链路

### 1. 患者端链路

`身份核验/建档 -> 判断是否已有有效排队 -> 创建 visit -> 到诊 -> 生成 assessment -> 自动入队 -> 返回队列结果`

### 2. 人工分诊链路

`到诊 -> 分诊评估 -> 自动入队 -> 工作台叫号 -> 完成接诊`

### 3. 异常治理链路

`识别 TRIAGED 但没有活跃票据的记录 -> 后台异常列表 -> 管理员补录 -> 重新回到主链路`

---

## 必背状态

### Visit 状态

`REGISTERED -> ARRIVED -> TRIAGED -> QUEUING / IN_TREATMENT -> COMPLETED`

### Queue 状态

`WAITING -> CALLING -> MISSED / COMPLETED`

### 展示层补充

- `QUEUEING`：排队中
- `WAITING_FOR_CONSULTATION`：候诊中

一句话解释：

> visit 是就诊生命周期，queue 是排队处理流程。

## 高频问答速记

### 1. 核心难点是什么？

跨模块状态一致性，不是单个接口难。

### 2. 为什么不用微服务？

当前阶段先保证闭环和一致性，模块间耦合强，拆微服务成本太高。

### 3. 为什么不是纯先来后到？

医疗场景要考虑分诊等级、特殊人群、等待老化和高峰策略。

### 4. AI 出问题怎么办？

直接回退到本地规则，不阻断主链路。

### 5. 为什么做异常治理？

因为真实业务里最怕中间状态不一致，系统必须可修复、可运营。

### 6. 多端数据怎么保持一致？

不是每端维护一套状态，而是共用统一的 visit / queue / assessment 模型。

### 7. Redis 做什么？

做队列调度、高频状态处理和叫号相关快速读写。

### 8. 你最想体现什么能力？

不是写了多少页面，而是把真实业务闭环、状态流转和异常治理一起做完整。
