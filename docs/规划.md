# 患者智能排队分诊系统扩展与 AI 改造规划

## Summary
- 现状基线：项目已具备 `认证/患者/就诊/分诊/排队/看板` 主链路，技术栈是 `Spring Boot 3 + MyBatis-Plus + Redis + Vue 3`，后端编译通过、前端构建通过，核心单元测试可跑通；完整 `mvn test` 依赖 Docker 中的 Testcontainers Redis。
- 当前短板：分诊仍是“规则+阈值”驱动，预计等待时长仍是“固定每人 5 分钟”，前端没有把“分诊完成 -> 正式入队”闭环接起来，大屏和后台不是实时推送，权限仍是登录服务内硬编码而非数据库驱动。
- 目标收敛：第一阶段做“可展示的智能分诊”，采用云端 AI API，落点是“护士分诊台 + 管理后台”，AI 只给建议，不直接落正式结果，必须由护士确认。

## Extension Checklist
- `P0` 业务闭环补全：补上“分诊确认后选择诊室并创建排队票据”，并在后台显示“已分诊未入队”异常。
- `P0` 实时化：为后台、工作台、大屏补 `SSE/WebSocket` 推送，覆盖叫号、候诊变化、超时告警和大屏刷新。
- `P0` 平台治理：把角色权限改成数据库驱动，增加用户、角色、科室、诊室的管理能力。
- `P1` 患者服务：预约/预检、移动端排队查询、自助建档、短信或广播通知、排队小票打印。
- `P1` 运营分析：日报周报、诊室吞吐、拥堵趋势、高优先级滞留、导出报表。
- `P1` AI 数据底座：补“分诊修正原因、最终接诊科室、医生反馈、结果标签”，为后续模型优化做回流。
- `P2` 平台扩展：多院区/多楼层、HIS/LIS/EMR 对接、脱敏审计、知识库问答与患者助手。

## AI V1 Implementation
- 后端新增 `modules.ai` 模块，分为 `controller/service/provider/prompt/dto/vo`；Provider 层做成可配置云端适配器，使用 `base-url/api-key/model/timeout/retry`，业务层不直接绑定某一家厂商 SDK。
- 新增表 `ai_triage_record`，保存 `visitId`、患者/就诊快照、生命体征输入、规则结果、AI 建议、置信度、风险标记、解释文本、provider/model、耗时、状态（`SUGGESTED/CONFIRMED/REJECTED/FAILED`）、确认人、关联 assessmentId。
- 扩展 `triage_assessment`，增加 `decision_source`、`ai_record_id`、`ai_confidence`、`ai_rationale`、`risk_flags`，保留现有规则引擎作为降级路径。
- 新增接口 `POST /api/ai/triage/suggestions`：输入 `visitId + symptomTags + vitals + manualNotes`，输出 `ruleSuggestion`、`aiSuggestion`、`confidence`、`riskFlags[]`、`rationale`、`aiRecordId`。
- 扩展 `POST /api/triage/assessments` 与重评估 DTO：新增 `decisionSource`（`RULE/AI/MANUAL`）、`aiRecordId`、`overrideTriageLevel`、`overrideRecommendDeptId`、`overrideFastTrack`。当 `decisionSource=AI` 时，以 AI 建议为基线，再叠加护士修正。
- 新增诊室查询接口 `GET /api/clinics/depts/{deptId}/rooms`。分诊确认后在同页提供“确认并入队”；默认诊室规则固定为：若当前账号绑定诊室且属于推荐科室，则默认该诊室，否则默认该科室第一个启用诊室。
- 护士分诊页改造成“双轨对比”界面：左侧规则结果，右侧 AI 结果，中间显示置信度、风险标记、解释摘要，操作流固定为 `采纳 AI / 保持规则 / 人工调整 -> 确认保存 -> 选择诊室入队`。
- 管理后台新增 AI 面板：展示 `AI 采纳率`、`规则与 AI 冲突数`、`高风险待处理人数`、`已分诊未入队人数`、最新高风险建议列表。第一阶段不做患者端聊天，也不在大屏直接展示 AI 交互。

## Public Interfaces And Data
- 保持现有 `auth/patient/visit/queue/dashboard` 主接口兼容，只对分诊相关 DTO/VO 做向后兼容扩展。
- 新增后端/前端类型：`AiTriageSuggestionRequest`、`AiTriageSuggestionVO`、`RuleSuggestionVO`、`AiRiskFlagVO`、`AiTriageRecordVO`、`ClinicRoomOptionVO`。
- 前端 `TriageAssessment` 类型增加 AI 来源与覆盖字段，排队模块增加诊室选项结构用于“确认并入队”。
- 日志策略固定为：DB 中保存 AI 审计记录；应用日志只保留摘要与耗时，不打印完整敏感文本；后台默认展示脱敏患者标识。
- 失败策略固定为：AI 超时、解析失败或返回异常时，页面明确提示“AI 不可用”，并自动退回现有规则分诊流程，不允许阻塞人工分诊。

## Test Plan
- 后端单测：Prompt 组装、AI 返回解析、规则降级、`decisionSource=AI` 的 assessment 保存、默认诊室选择、assessment 到 queue 的闭环。
- 后端集成：在 Service 边界 mock AI Provider，不依赖外网；保留现有 Redis/Testcontainers 的排队生命周期测试。
- 前端验证：`npm --prefix web run typecheck`、`npm --prefix web run build`、护士分诊确认流、后台 AI 面板渲染、AI 失败降级提示、脱敏展示。
- 手工验收场景：
  1. 护士打开就诊记录，拿到规则+AI 建议，采纳 AI，确认后成功入队。
  2. 护士拒绝 AI，改成人工结果，系统能完整留痕。
  3. 管理员能看到高风险待处理、AI 采纳率、规则冲突和未入队异常。
  4. AI 服务超时或返回异常时，人工分诊仍能继续完成。

## Assumptions
- 第一阶段以演示和课程展示为主，优先做“看得见的智能闭环”，不是生产级全量替换。
- AI 只做建议，不自动落正式分诊结果。
- AI 使用云端接口，但通过 Provider 抽象保留后续切换空间。
- 第一阶段界面范围固定为“护士分诊台 + 管理后台”，大屏和患者侧只消费下游结果，不直接接 AI。
- 若时间紧，实时推送和数据库驱动权限可与 AI 主链路并行设计、分阶段落地，但“AI 建议 -> 人工确认 -> 入队闭环”必须作为首批完成项。
